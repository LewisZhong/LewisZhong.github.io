MySQL数据库基础考察点

发表于 2018-03-27 | 分类于 数据库
字数统计: 729 | 阅读时长 ≈ 3

数据类型

  1. 实数类型：FLOAT,DOUBLE,DECIMAL

    DECIMAL可以存储比GIgINT还大的整型，可以用于存储精确的小数

  2. 字符串类型：VARCHAR,CHAR,TEXT,BLOB

    VARCHAR类型用于存储可变长度的字符串，比定长类型更节省空间；VARCHAR使用1或2个额外字节记录字符串长度；如果存储长度超出指定长度，会被截断。
    CHAR是定长的；CHAR会根据需要采用填充以方便比较；CHAR适合存储很短的字符串，或者所有者接近于同一个长度；超出设定长度也会被截断
    对于经常变更的数据，char比varchar更好，因为char不容易产生碎片；非常短的列，char比varchar在存储空间上更有效率

  3. 日期时间类型

    尽量使用TIMESTAMP,比DATETIME空间效率更高

  4. 列属性

    auto_increment, default, not null, zerofill

MySQL基础操作

  1. MySQL的连接和关闭

    mysql -u -p -h -P,分别代表数据库的用户名，密码，主机号，端口号

MySQL数据表引擎

  1. InnoDB表引擎（优先）

    1.1 默认事务性引擎，最重要最广泛的存储引擎
    1.2 数据存储在共享表空间
    1.3 对主键的查询性能高于其他类型的存储引擎
    1.4 从磁盘读取数据时自动在内存构建hash索引
    1.5 插入数据时自动构建插入缓冲区
    1.6 支持崩溃后的安全恢复，支持热备份，支持行级锁，支持外键

    1. MyISAM表引擎
      2.1 5.1版本前是默认的存储引擎
      2.2 拥有全文索引、压缩、空间函数
      2.3 不支持事物和行级锁，不支持崩溃后的安全恢复
      2.4 表存储在两个文件，MYD和MYI
    2. 其他表引擎
      Archive,Blackhole,CSV,Memory

MySQL锁机制

  1. 读锁（共享锁）：共享的，多个用户可以同时读一个资源
  2. 写锁（排他锁）：排他的，一个写锁会阻塞其他的写锁和读锁
  3. 锁粒度

    表锁，系统性能开销最小，会锁定整张表，MyISAM使用表锁
    行锁，最大程度地支持并发处理，但是也带来了最大的锁开销，InnoDB支持行锁

MySQL事务处理

  1. 提供事务处理的表引擎：InnoDB
    同一事务中，应尽量使用同一种表引擎

存储过程

为以后的使用而保持的一条或多条MySQL语句的集合，可以在存储过程中
创建表，更新数据，删除等

触发器

使用场景

  1. 可通过数据库中的相关表实现级联更改
  2. 实时监控某张表中的某个字段的更改而做出相应的处理
  3. 某些业务编号的自动生成
  4. 滥用会造成数据库及应用程序的维护困难
